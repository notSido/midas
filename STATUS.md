# STATUS

This file lists **only** capabilities backed by a reproducing artifact
(a green `cargo test`, a captured CLI invocation, a decodable trace, or a file a
third-party tool can open). No forward-looking or aspirational claims appear
here. If a capability is not listed, it is not implemented.

Replay evidence labelled **checkpoint** records the production tree when that
capability slice landed. The newest capability row and the "Not yet
implemented" section describe the latest production frontier.

## Not yet implemented

The Win64 environment is only **partially** implemented. The earlier 8-call wall,
its VM-slot diagnosis, and the bounded in-image-stub experiment remain historical
evidence in `docs/FINDINGS-M3-import-wall.md`; that experiment did not advance the
observable frontier and did not prove candidate 1 causally false.

The bounded `Sleep` slice advances formal sample 1 through call 39 after
`CreateThread`, `GetProcessHeap`, and `RtlAllocateHeap`, then exposes a stable
polling loop: every immediate `Sleep(1)` return leads back to the same named API
after 3,527 executed instructions. A 200-call replay contains the preserved
38-call prefix plus 162 `Sleep` calls and stops only at `max_calls`. This is an
execution bound, not evidence that guest time elapsed. A disposable formal
trace identifies the repeated branch as a zero comparison against rawless
`.themida` byte `0x000000014005aae0`; changing only that byte to one avoids the
next Sleep. An isolated direct-entry control for the recorded child handles
`LoadLibraryA`, `GetProcAddress`, and the fixed-zero `timeGetTime` policy, then
still reaches the unresolved zero target without accessing or changing that
byte; exact CPU-context restoration leaves the main loop unchanged. This
falsifies only the tested single child run as the loop release, not all possible
scheduling or Windows thread-start models. Sample 2 retains its separate
indirect call-through-zero after call 33. Sample 3 corroborates the same polling
shape from its separate call 38 through the 200-call bound. Samples 2 and 3 are

CPU contexts are now available as an emulator primitive, but no production path
uses them to schedule or execute a created thread. They do not allocate or
snapshot guest memory, a stack, TEB, TLS, `Win64Env`, or virtual time. The fixed
`timeGetTime` result is not an advancing clock. Created threads therefore remain
records only, and `Sleep` does not advance a clock or yield to them. DLL thread
notifications, suspend/resume, termination, waits/signaling, handle close/reuse,
detailed security/inheritance
behavior, and last-error state remain unmodeled. The observed start address is
a downstream thread-entry candidate, not an OEP finding. Guest
manifest/version-policy discovery and other version APIs are unmodeled.
Vectored-handler callback invocation, removal, exception dispatch, lifetime,
and concurrency also remain unmodeled. General current-directory mutation and
fuller filesystem semantics, non-NULL module filename lookup, TEB `ClientId`,
`RtlFreeHeap`, and `RtlReAllocateHeap` also remain unmodeled; no module/handle
lifecycle or reallocation semantics are claimed.
`examples/run_loader.rs` reproduces the current wall, and
`docs/FINDINGS-M3-import-wall.md` records the full finding chain.

Also not implemented (per `docs/CHARTER.md`): OEP detection, trace recording, VM
detection, and the IR lifter. None has a passing acceptance artifact yet.
