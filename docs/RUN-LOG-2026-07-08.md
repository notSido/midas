## Crash analysis

### What ran

- Entry point: `0x14030b058` (`.boot` section, RVA `0x30b000`).
- Execution stayed in `.boot` for all 26.5M instructions.
- 199 unique addresses, 5 hottest all in `0x14030b07e`â€“`0x14030b1a5`.
- **Zero WinAPI calls.** Verbose run showed only the 30 API
  registrations at startup, then the crash â€” no `VirtualAlloc`,
  `GetProcAddress`, `NtQuery*`, nothing. The boot loader never reached
  the API-call stage.

### The hot loop is aPLib decompression

Disassembly of `.boot` at `0x14030b060`â€“`0x14030b1c0` (via capstone)
identifies the hot loop as the **aPLib decompression routine**:

```
0x14030b064  mov rsi, [r10+0x10]   ; source = compressed stream
0x14030b068  mov rdi, [r10+0x20]   ; destination = decompressed output
0x14030b06d  mov dl, 0x80          ; bit buffer init
0x14030b07e  add dl, dl            ; << HOT: shift bit, CF=top bit
0x14030b080  jne 0x14030b089       ; << HOT: skip refill if buffer nonzero
0x14030b082  mov dl, [rsi]; inc rsi; adc dl, dl  ; refill from stream
0x14030b089  jae 0x14030b06f       ; << HOT: CF=0 â†’ literal byte copy
; CF=1 â†’ match token: read offset + length, rep movsb
```

The repeated `00 d2 75 07 8a 16 48 ff c6 10 d2` pattern
(`add dl,dl` / `jnz` / `mov dl,[rsi]` / `inc rsi` / `adc dl,dl`) is
the textbook aPLib bit-stream decoder. Themida uses aPLib for its
compression stage.

Context struct pointer: `r10`. Source: `[r10+0x10]`. Dest: `[r10+0x20]`.
Mirror-sync writes propagated: **2,882,229** â€” so decompression was
actively writing output before the crash.

### The crash

```
READ_UNMAPPED at 0x2801984b6 (size: 1)
Stopped at RIP: 0x4d091
Crash-site bytes at RVA 0x4d091: d0 04 00 00 00 00 00 00 ...
```

- RIP `0x4d091` is in the **low-memory mirror** of `.idata`
  (RVA `0x4d000`, mapped at `0x0`â€“`0x1000000` for RVA addressing).
  Themida jumped to an RVA-as-pointer (no image base added).
- Bytes `d0 04 00` decode as `shl byte ptr [rax+rax], 1`.
- `rax` = `0x1400cc25b` (inside `.themida`, RVA `0xcc25b`).
- Effective read = `rax*2` = `0x2801984b6` (unmapped â€” 2Ă— a `.themida`
  address, far above the image).

So: execution **left the aPLib loop** and jumped to `0x4d091`
(low-mem `.idata`), where it tried to execute data bytes.

### OEP detection did not fire

`Themida bounds set: 0x14004d000 - 0x14004e000` â€” the OEP detector's
"Themida bounds" covers only `.idata` (4 KB). The crash at `0x4d091`
fell **inside** these bounds (it's in `.idata`'s low-mem mirror at
`0x4d091`), so the "leave Themida bounds = OEP" heuristic never
triggered. Result: `Error: OEP not found`.

## Root-cause hypotheses (not yet confirmed)

Ranked by likelihood:

1. **Bad jump target out of the aPLib loop.** The loop has no explicit
   exit; something did an indirect jump/call to an RVA (`0x4d091`)
   that landed in `.idata`'s low-mem mirror. Need a RIP ring buffer to
   see the instruction that jumped. The jump target `0x4d091` looks
   like an unresolved IAT/ILT thunk RVA â€” possible the boot loader
   tried to call an import whose thunk was never resolved.

2. **`r10` context struct mis-set.** `rsi`/`rdi` come from `[r10+0x10]`
   / `[r10+0x20]`. If the code that builds the `r10` struct before
   `0x14030b060` depended on PEB/TEB or TLS values the unpacker didn't
   provide correctly, the source/dest could be subtly wrong, leading
   the decompressor to corrupt memory and then jump to garbage.

3. **TLS callbacks not run.** The protection log said
   "Processing TLS . . . OK". Themida may use TLS callbacks that must
   execute **before** the entry point. If the unpacker jumps straight
   to the entry point without running TLS callbacks, boot-loader
   initialization that callbacks would have done is missing. Need to
   check whether `unpacker.rs` runs TLS callbacks at all.

4. **Themida bounds detection too narrow.** `find_themida_section()`
   matched only `.idata` (4 KB) as the Themida section, missing
   `.themida` (2.75 MB, no raw data) and `.boot` (1.5 MB). This doesn't
   cause the crash but means OEP detection would misfire even if
   execution reached the real OEP â€” `.boot` and `.themida` are outside
   the bounds, so any transition between them would falsely read as
   "OEP reached." Separate bug from the crash but worth fixing.

## Next steps (when picking this back up)

1. **Add a RIP ring buffer** (last ~32 addresses) to the code hook,
   dump on crash. This directly reveals the instruction that jumped
   to `0x4d091`. Edit in `src/unpacker.rs` around the code hook
   (~line 223) and the `READ_UNMAPPED` handler (~line 506).
2. **Check TLS callback handling** in `unpacker.rs` â€” if absent, add
   it. The `.tls` section is at RVA `0x4e000`; the TLS directory may
   point to callbacks that must run first.
3. **Fix `find_themida_section()`** to also match `.boot` and
   `.themida` (sections with no raw data or `virtual_size >> raw_size`
   should be candidates, not just `.idata`).
4. **Log `r10`/`rsi`/`rdi`** when execution first hits `0x14030b060`
   (the aPLib entry) to verify the context struct is sane.
5. **Re-run after each fix** â€” the crash is deterministic and fast
   (~5s to reach 26M instructions), so iteration is cheap.
